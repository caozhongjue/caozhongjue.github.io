性能测试，自动化测试，接口测试，app测试

发表于 2020-08-01 | 分类于 性能测试

app测试

一、app测试点
功能测试、用户体验、兼容性测试、ui测试、交叉事件、安装卸载升级、性能测试、安全测试
二、app兼容性测试要考虑哪方面
1、主流手机型号
2、不同的操作系统和版本（ios考虑10版本以上，android 6-10）
3、不同的手机分辨率和手机尺寸、屏幕
4、不同的手机配置
三、兼容可能存在哪些问题
1、ui界面问题
2、部分功能不能使用
四、app和web测试的区别
1、系统架构方面：
web项目是b/s架构的基于浏览器，app项目是c/s架构，必须要安装客户端，web只要更新服务端，客户端会同步更新，app项目则客户端和服务端都要更新
2、性能方面：
web页面主要关注响应时间，app则需要关注流量，电量,cpu,memory，这些
3、兼容性方面：
web是基于浏览器，所以更倾向于浏览器和电脑硬件，电脑系统的兼容，app测试则要看分辨率，屏幕尺寸，还要看设备系统。web测试是基于浏览器的所以不必考虑安装卸载。app是客户端，则必须测试安装、更新、卸载、除了常规的安装更新、卸载还要考虑到异常场景包括安装的中断、弱网、安装后删除文件
五、App测试中ios和Android的区别
1、Android长按home键呼出应用列表和切换应用，然后右滑则终止应用
2、多分辨率，android分辨率比较多，20多种，ios比较少
3、手机操作系统，android比较多，ios比较少且不能降级
4、操作系统习惯，android back键是否被重写，测试试点击back键后的反馈是否正确，应用数据从内存移动到sd卡后能否正常进行
5、push测试，android点击home键程序后台运行时此时收到push，点击后呼醒应用，此时是否正确跳转到app页面，ios点击home键关闭程序和屏幕锁屏的情况
6、安装测试，android的下载和安装工具渠道比较多，ios主要有appstore itunes
7、升级测试，可以被升级的必要条件，新旧版本具有相同的签名，新旧版本具有相同包名，一个标示符区分新旧版本
六、adb 工具
android debug bridge （安卓调试桥）它就是命令工具窗口，用于通过电脑端与模拟器或真机设备交互
1、常用命令
adb connect 127.0.0.1:620001(连接虚拟机，真机直接用设备名)
adb devices 查看设备
adb install D:/xxx/xxx/xxx/xxx.apk 安装apk文件
adb install -r D:/xxx/xxx/xxx/xxx.apk 覆盖安装apk文件
adb -s 设备名 install D:/xxx/xxx/xxx.apk 指定安装apk文件到某设备中
adb uninstall 包名.apk (包名查询 adb logcat |findstr START)
adb shell “dumpsy window | grep inCurrentFocus” 查看包名
adb logcat -v time 查看手机日志
adb logcat -c 清除设备日志
adb shell 进入shell模式 android底层是linux，linux命令在此也可用
adb push D:/xxx/xx/ 安卓设备文件（/sdcard/xxx文件）
monkey介绍：
monkey是每个手机自带的，通过adb去使用它，monkey测试是随机操作
adb shell monkey -p 包名 -v 1000（次数）
app性能测试的流程：定制app手机端的性能测试
1、首先使用itest工具监控app的一些资源使用情况
2、在doc窗口输入monkey的命令执行一段时间，延时
3、跑完monkey查看itest监控的图表信息
4、根据图表绘制成测试报告
adb shell monkey -p 包名 –throttle 300 -v 次数
adb忽略异常停止命令
–ignore-crashes 忽略崩溃
–ignore-timeout 忽略超时
–ignore-security-exception 忽略异常
monkey如何停止
adb shell
ps -ef|grep monkey
kill -9 pid号
app稳定性测试流程
adb logcat -c 请除设备上的日志
adb logcat -v time > D:/logcat.log 输出logcat日志到本地
adb shell monkey -p 包名 –throttle -300 -v 次数 > D:/monkey.log
当monkey停止，先查看monkey日志，看有没有跑完设定的次数，没跑完就查看logcat日志，如是app崩溃，看有没有出null pointer空指针异常，如出现就截图发给开发，如是无响应，就搜索anr关键字，找到造成anr进程号，把手机设备上的tract.txt文件导出到本地，截图anr关键日志和tract.txt文件一并发给开发

ui自动化

一、用于回归测试，主要是验证之前版本的功能是否出现问题
不是所有项目都适合做自动化测试
条件：
1、迭代多
2、研发周期长（至少半年以上）
3、界面元素稳定

ui自动化怎么做
python+selenium+unittest+jenkins
二、ui自动化环境搭建
1、安装python(很简单，先去官网上下载下来，傻瓜式安装，配置环境变量，在cmd窗口输python验证)
2、安装selenium (命令： pip install seleniu (前提把python的Scripts文件夹添加到环境变量)) pip list验证是否成功
3、安装浏览器对应的驱动 如是Chrome就搜谷歌浏览器对应的版本，把下载下来的驱动解压放到python安装目录下（和python.exe文件同目录）
4、验证，写脚本看是否启动浏览器

from selenium import webdriver
import time
#import winsound.Beep as Beep
#指定是chrome的驱动
#执行到这里的时候selenium回去到指定路径将chrome driver程序运行起来
driver = webdriver.Chrome()#返回一个webdriver对象
#隐式等待是通过一定的时长等待页面上某元素加载完成。如果超出了设置的时长元素还没有被加载，则抛出NoSuchElementException异常，如果在设置时间内找到某元素，会马上执行后面的操作；
driver.implicitly_wait(10)
#get方法。打开指定网址
driver.get('https://search.51job.com')

三、selenium定位方式

<input name="user_name" id="username" type="text" autocomplete="off" placeholder="账号" class="el-input__inner">
<input id="passwd" type="password" autocomplete="off" placeholder="密码" class="el-input__inner">    
<div class="el-form-item__content"><button type="button" class="el-button el-button--primary el-button--medium" style="width: 100%;"><!----><!----><span><span>登 录</span></span></button><!----></div>
<a href="www.baidu.com">赶紧百度一下吧</a>

dr.find_element_by_id('username')   #通过id定位
dr.find_element_by_name('user_name')  #通过name定位
dr.find_element_by_tag_name('input') #通过tag元素名称定位
dr.find_element_by_link_text('百度一下')  #通过链接名定位
dr.find_element_by_class_name('el-input__inner')
dr.find_element_by_css_selector('.passwd')或dr.find_element_by_css_selector('.el-input__inner')  #css的定位如是id 就用#号，class就用.号
dr.find_element_by_xpath('.//*input[@name="user_name"]')  #用属性定位
dr.find_elements_by_partial_link_text('赶紧百度一下吧')

css_selector()和xpath语法区别
1、css相对快一点，稳定灵活点
2、语法不一样
css前面不需要//,属性前不需要@，属性不能用and,层次用>,同层次不需要写位置
xpath前面需要//,属性前要@,属性之可用and ,层次用/,同层次需要写位置
